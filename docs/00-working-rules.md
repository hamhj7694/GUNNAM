# 개발 작업 규칙

## 기준 문서 우선순위

1. `PRD.md`: 제품 범위, 사용자 흐름, 기능 및 인수 조건의 최상위 기준
2. `docs/01-product-scope.md`: 구현 범위와 금지 범위
3. `docs/02-architecture-and-state.md`: 라우팅, 상태, 파일 책임
4. `docs/03-ui-and-content-rules.md`: UI, 반응형, 문구 및 표시 규칙
5. `docs/04-testing-and-done.md`: 검증 방법과 완료 정의
6. `TODO.md`: 현재 개발 순서와 실제 진행 상태

문서가 충돌하면 위 순서를 따른다. PRD 변경으로 하위 문서가 달라져야 하면 같은 작업에서 관련 문서와 `AGENTS.md`, `TODO.md`도 함께 갱신한다.

## 매 작업 시작 전

- `PRD.md`에서 작업 대상 요구사항과 범위 밖 항목을 확인한다.
- `AGENTS.md`의 기능별 파일 매핑으로 수정 범위를 정한다.
- 관련 `docs` 문서의 규칙과 인수 조건을 확인한다.
- `TODO.md`에서 작업 상태와 선행 작업을 확인한다.
- 실제 파일 구조가 문서와 다르면 코드를 우선 조사하되, 제품 의도는 PRD를 따른다.

## 구현 원칙

- MVP 범위를 임의로 확장하지 않는다.
- 상태는 React 메모리 상태로만 관리한다.
- 서버, API, DB, 인증, 공유 링크, 영속 저장을 추가하지 않는다.
- 파생 표시값은 원본 상태에 중복 저장하지 않고 유틸에서 계산한다.
- 이미지가 없을 때 placeholder나 빈 공간을 남기지 않는다.
- 수락과 거절의 상태 및 동작을 독립적으로 유지한다.
- 사용자 노출 한글은 UTF-8로 저장하고 깨짐 여부를 확인한다.
- 기존 동작을 변경하면 관련 인수 조건과 회귀 항목을 검증한다.

## 변경 관리

- 한 작업은 하나의 명확한 사용자 가치 또는 기술 기반을 완성하도록 작게 나눈다.
- 기능 구현과 필요한 스타일, 접근성 속성, 검증을 함께 처리한다.
- 구현이 끝난 뒤에만 `TODO.md` 항목을 `[x]`로 바꾼다.
- 일부만 끝난 작업은 완료 처리하지 않고 하위 항목으로 나눈다.
- 계획이 바뀌면 이유를 `TODO.md`의 결정 기록에 짧게 남긴다.
- 관련 없는 사용자 변경은 수정하거나 되돌리지 않는다.

## 금지 사항

- 요청 없이 범위 밖 기능을 추가하지 않는다.
- ShowCardPage에 편집, 공유, 설정 등 응답자의 선택을 방해하는 UI를 넣지 않는다.
- 거절 사유 분류나 다시 묻기 흐름을 만들지 않는다.
- `acceptResultText` 또는 `rejectResultText`가 비었을 때 fallback 규칙을 제거하지 않는다.
- 이미지에 강제 crop을 적용하지 않는다.
- 테스트하지 않은 항목을 완료로 표시하지 않는다.
