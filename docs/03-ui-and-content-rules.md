# UI, 반응형, 콘텐츠 규칙

## 레이아웃

- 앱 배경 위 중앙에 최대 430px 너비의 모바일 프레임을 둔다.
- 데스크톱 프레임에는 둥근 모서리, 옅은 테두리, 그림자를 사용한다.
- 420px 이하에서는 프레임 장식을 제거하고 전체 화면을 사용한다.
- 최소 320px 폭에서 가로 스크롤이나 주요 요소 겹침이 없어야 한다.

## 색상 맥락

- 기본 및 메인 행동: 녹색
- 수락 결과 및 수락 작성 맥락: 파란색
- 거절 결과 및 거절 작성 맥락: 붉은색
- 거절 버튼은 수락 버튼보다 시각적 강도가 낮아도 선택 가능성을 해치지 않아야 한다.

## 카드와 텍스트

- 메인 카드와 결과 카드는 모바일 화면의 핵심 초점이어야 한다.
- 긴 문구는 카드 폭 안에서 안전하게 줄바꿈한다.
- 사용자가 입력한 줄바꿈은 결과 표시에서도 보존한다.
- 결과 카드와 실제 결과 화면은 동일한 콘텐츠 규칙을 사용한다.
- 버튼 문구가 길어도 잘리거나 레이아웃을 깨지 않게 한다.

## 이미지

- 이미지가 없으면 DOM 요소, placeholder, 고정 높이 빈 공간을 남기지 않는다.
- 메인 및 결과 이미지는 `object-fit: contain`을 기본으로 한다.
- 이미지 선택 입력은 `accept="image/*"`를 사용한다.
- 이미지는 FileReader data URL로 React 상태에 반영하고 카드 데이터와 함께 localStorage에 저장한다.
- 값이 있을 때만 재선택과 삭제 상태를 명확히 표시한다.

## 입력과 검증 문구

- 빈 수락 버튼 fallback: `좋아요`
- 빈 거절 버튼 fallback: `싫어요`
- 빈 결과 문구: 해당 버튼의 최종 표시 문구
- 메인 문구 필수 오류: `상대에게 보여줄 내용을 적어주세요.`
- 답장 라벨: `하고 싶은 말을 남겨주세요`
- 빈 답장 오류: `메시지를 입력해주세요.`
- placeholder는 힌트일 뿐 저장값이나 fallback 값으로 취급하지 않는다.

## 상호작용과 접근성

- 모든 버튼에 `type="button"`을 명시한다. 폼 제출 버튼만 `submit`을 사용한다.
- 아이콘만 있는 뒤로가기 버튼에 목적을 설명하는 `aria-label`을 제공한다.
- 토글은 현재 상태를 `aria-pressed` 등으로 노출한다.
- 파일 input을 시각적으로 숨기더라도 연결 버튼으로 키보드 조작이 가능해야 한다.
- focus 표시를 제거하지 않고 각 색상 맥락에서 식별 가능하게 한다.
- 작성 미리보기의 답장 폼은 실제 입력·제출이 되지 않아야 한다.
- 결과 페이지 우측 상단의 답변 초기화는 원형 버튼으로 표시하고 결과 맥락 색상과 명확한 접근성 이름을 사용한다.
- 히스토리 카드는 받은 답변을 가장 크게 표시하고 질문은 펼칠 수 있는 보조 맥락으로 숨긴다.
- 히스토리 삭제 버튼에는 삭제 대상을 설명하는 접근성 이름과 확인 절차를 제공한다.
