# 온라인 확장 단계별 구현 계획

- 상태: 설계 초안
- 작성일: 2026-07-15
- 기준: `docs/06-online-expansion-rfc.md`
- 목적: 온라인 확장을 직접 건넴의 회귀 없이 작은 출시 단위로 구현하기 위한 실행 순서와 품질 게이트 정의

## 1. 총괄 원칙

- 현재 `/create → /show → /result/* → /history` 흐름은 `직접 건넴` 전용으로 보존한다.
- 온라인 상태와 현재 localStorage 상태를 `App.jsx`의 동일 객체에 합치지 않는다.
- 프론트엔드와 백엔드는 승인된 OpenAPI 계약과 fixture를 기준으로 병렬 개발한다.
- 실제 서버·DB·스토리지·인증 공급자 생성과 배포는 별도 D5 승인을 받는다.
- `모두의 건넴`은 운영·안전 기능을 먼저 갖춘 뒤 활성화한다.
- 각 단계는 해당 품질 게이트를 통과해야 다음 단계로 넘어간다.

## 2. 단계 개요

```text
P0 정책·계약 확정
  → P1 직접 건넴 회귀 보호와 방식 선택
  → P2 비회원 링크 MVP 수직 슬라이스
  → P3 링크 관리와 공개 범위
  → P4 이미지
  → P5 선택적 계정 귀속
  → P6 링크 MVP 보안 강화·출시
  → P7 운영·안전 기반
  → P8 모두의 건넴
```

## P0. 정책·계약 확정

### 목표

프론트엔드와 백엔드가 독립 구현할 수 있도록 제품 정책, API, 권한과 테스트 기준을 고정한다.

### 작업

- 승인된 RFC 내용을 PRD와 하위 규칙 문서에 반영한다.
- 미결정 항목에 Decision ID와 승인 상태를 부여한다.
- `contracts/openapi.yaml` 초안을 만든다.
- 성공·오류·권한별 JSON fixture를 정의한다.
- 카드·답변·토큰 위협 모델과 권한 행렬을 작성한다.
- 직접 건넴 회귀 시나리오와 온라인 품질 추적표를 만든다.
- 온라인 기능 플래그 전략을 정한다.

### 완료 조건 G0

- 모든 요구사항에 구현 위치와 테스트가 연결된다.
- D2·D3·D4·D5 결정이 문서에서 구분된다.
- 응답용·관리용·수정용 토큰의 권한이 분리된다.
- 온라인 기능이 비활성화된 상태에서 기존 앱 빌드와 직접 흐름이 정상이다.

## P1. 직접 건넴 회귀 보호와 방식 선택

### 목표

현재 기능을 보존하면서 새 카드 작성 전에 건네는 방식을 선택할 수 있게 한다.

### 프론트엔드 작업

- `/create/mode` 방식 선택 페이지 추가
- `DeliveryModeCard` 추가
- 홈의 `새 카드 만들기`는 초기화 후 `/create/mode`로 이동
- 홈의 `카드 이어서 만들기`는 기존 `/create` 유지
- `직접 건넴`은 기존 `/create`로 연결
- 온라인 기능 준비 전 `링크로 건넴`, `모두에게 건넴`은 기능 플래그 아래에서 준비 상태로 표시
- 사용자가 직접 조정한 홈 버튼 순서와 CSS 클래스를 보존

### 테스트 기반

- Vitest와 React Testing Library 도입 여부 확정
- 홈 새 카드·이어 만들기·히스토리 회귀
- localStorage 복구와 초기화
- 3단계 작성, 보여주기, 수락·거절, 답변 초기화와 로컬 히스토리
- 직접 모드에서 네트워크 요청이 발생하지 않는 테스트
- 320px·390px·데스크톱과 키보드·스크린리더 검증

### 완료 조건 G1

- 직접 건넴의 현재 기능과 로컬 키가 유지된다.
- 서버가 없어도 직접 건넴이 완전히 동작한다.
- 온라인 미구현 기능이 실제 공유 성공처럼 보이지 않는다.

## P2. 비회원 링크 MVP 수직 슬라이스

### 목표

텍스트 카드 하나를 로그인 없이 온라인으로 만들고, 선택 답변을 받은 뒤 관리 링크에서 확인하는 전체 경로를 완성한다.

### 포함 범위

```text
텍스트 카드 생성
→ 응답 링크·비밀 관리 링크 1회 발급
→ 응답 링크 카드 조회
→ 수락/거절 필수 + 추가 의견 선택 제출
→ 관리 링크에서 받은 답변 목록 확인
```

### 제외 범위

- 이미지
- 계정 귀속
- 닉네임이 미결정이면 nickname 필드 UI
- 전체 답변 공개 전환
- 답변 수정
- 카드 마감
- 링크 재발급
- 모두의 건넴

### 프론트엔드 작업

- `src/api/` 온라인 API adapter와 mock/HTTP 구현 분리
- `src/features/online/`에 온라인 draft와 페이지 분리
- `/online/create`
- `/c/:shareToken`
- 관리 토큰 bootstrap 후 `/manage/...` 관리 화면
- `OnlineResponseForm`: 수락·거절 필수, 추가 의견 nullable
- `OwnerResponsesPage`: 추가 의견 없는 답변도 빈 영역 없이 표시
- 로딩, 제출 중, 재시도, 404, 409, 410, 422, 429 상태

### 백엔드 작업

- 카드·답변 스키마와 forward migration
- 비회원 카드 생성과 토큰 서비스
- 관리 토큰을 관리 세션으로 교환
- 공개 카드 전용 DTO
- 선택 답변 제출과 idempotency
- 작성자 관리 응답 목록
- 요청 제한, 입력 검증, 오류 envelope와 관측성

### 완료 조건 G2

- 선택만 제출해도 서버에 답변 1건이 생성된다.
- 추가 의견의 빈 문자열은 `null`로 정규화된다.
- 선택 없이 추가 의견만 제출하면 기록이 생기지 않는다.
- 관리 토큰 원문이 DB·로그·분석·Referer에 노출되지 않는다.
- 관리 권한 없이 전체 답변에 접근할 수 없다.
- 재시도로 카드나 답변이 중복 생성되지 않는다.

## P3. 링크 관리와 답변 공개 범위

### 목표

비회원 작성자가 카드와 답변을 안전하게 운영한다.

### 작업

- `owner_only | counts_only | all_responses` 서버 projection
- 맞춤형 응답자 공개 안내
- 공개 동의 및 비동의 답변 처리
- 작성자 수동 마감·재개
- 받은 답변 개별 삭제
- 응답 링크 비활성화·재발급
- 익명 응답 수정 토큰과 기존 답변 수정
- 버전 충돌과 동시 수정 처리

### 완료 조건 G3

- `owner_only`는 비작성자에게 집계·닉네임·원문을 반환하지 않는다.
- `counts_only`는 수락·거절 숫자만 반환한다.
- `all_responses`는 공개 조건을 만족한 답변만 반환한다.
- 공개 범위를 줄이면 캐시를 포함해 즉시 비공개가 적용된다.
- 마감·링크 재발급·삭제가 원자적으로 동작한다.

## P4. 이미지

### 목표

현재 카드 경험을 유지하면서 온라인 이미지를 안전하게 저장한다.

### 작업

- 업로드 의도 발급, 객체 저장소 업로드, finalize 흐름
- MIME·크기·실제 디코딩 검증
- 메타데이터 제거와 안전한 파생 이미지
- 고아 파일 정리와 카드 삭제 연동
- 로딩·실패·재시도 UI

### 완료 조건

- data URL을 온라인 DB에 저장하지 않는다.
- 실패한 카드 생성이 고아 파일을 영구 방치하지 않는다.
- 파일 정책과 비용 한도가 승인돼 있다.

## P5. 선택적 계정 귀속

### 목표

비회원 카드 작성자가 원할 때 카드를 계정에 저장해 여러 기기에서 관리한다.

### 작업

- 인증 adapter
- 관리 권한과 로그인 계정의 원자적 claim
- 계정의 내 카드·받은 답변
- 중복 claim·탈취·네트워크 실패 처리
- 귀속 후 기존 관리 링크 처리

### 완료 조건 G4

- 유효한 관리 권한과 인증 계정이 모두 있어야 귀속된다.
- 동시에 한 계정만 귀속에 성공한다.
- 인증 실패 시 기존 비회원 카드 관리 상태가 손상되지 않는다.

## P6. 링크 MVP 보안 강화와 출시

### 작업

- 권한 음성 테스트와 침해 시나리오
- 백업·복원과 마이그레이션 리허설
- 요청 제한·스팸 방지
- 모니터링과 오류 대응
- 개인정보 안내, 삭제 요청 경로와 보관 정책
- 단계적 기능 플래그 활성화와 롤백

### 완료 조건

- 운영 정책과 인프라 D4·D5 승인을 받았다.
- staging 전체 흐름과 권한 테스트가 통과했다.
- 장애 시 신규 생성만 중단하고 기존 카드 조회를 유지할 수 있다.

## P7. 운영·안전 기반

`모두의 건넴`보다 먼저 구현한다.

### 필수 범위

- 신고 접수와 처리 상태
- 카드·작성자 차단 또는 숨김
- 스팸·자동 제출 방지
- 관리자 검토·숨김·제재
- 금지 콘텐츠 정책
- 문의·이의 제기
- 개인정보 열람·삭제
- 감사 기록, 모니터링과 롤백

### 완료 조건 G5

- 신고부터 처리 종료까지 추적된다.
- 차단과 삭제가 API·목록·검색·캐시에 전파된다.
- 운영 담당자와 처리 기준이 승인됐다.

## P8. 모두의 건넴

### 목표

로그인 작성자가 카드를 게시하고 누구나 공개 카드를 발견해 답한다.

### 초기 범위

- `/everyone` 모두의 건넴
- 로그인 작성자의 게시·게시 해제
- 최신순 탐색
- 최소 카테고리·검색은 별도 승인
- 답변 받는 중·마감 상태
- 공개 범위에 허용된 집계와 답변

### 범위 밖

- 개인화 추천
- 인기 랭킹
- DM, 매칭, 소개팅
- 팔로우와 고급 프로필

### 완료 조건

- `link` 카드는 탐색·검색·사이트맵에 나타나지 않는다.
- `public → link` 전환 시 탐색에서 제거되고 기존 공유 링크는 유지된다.
- 신고·차단·삭제가 공개 화면과 캐시에 즉시 반영된다.

## 3. 단계별 사용자 보고 기준

다음 상황에서는 구현을 멈추고 사용자에게 보고한다.

- 공개·개인정보·보관 정책을 확정해야 할 때
- 외부 벤더 도입, 비용, 계정과 비밀키가 필요할 때
- 운영 DB·스토리지 생성 또는 마이그레이션 직전
- 기존 직접 건넴의 동작 변경이 필요할 때
- 배포·도메인·실제 사용자 데이터 수집 직전
- 품질 게이트가 FAIL 또는 BLOCKED일 때
