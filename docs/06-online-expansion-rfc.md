# RFC: 건넴 온라인 확장 방향

- 상태: 초안(Draft)
- 작성일: 2026-07-15
- 목적: 현재 로컬 전용 건넴을 링크 공유와 공개 참여가 가능한 온라인 서비스로 확장하기 위한 제품·데이터 방향 정리
- 적용 범위: 향후 업데이트 제안이며, 승인 후 `PRD.md`와 개발 계획에 반영하기 전까지 현재 구현의 필수 요구사항은 아니다.

## 승인된 방향 기록

- 2026-07-15: `링크로 건넴`은 작성자 로그인 없이 생성할 수 있으며, 비밀 관리 링크로 관리하고 필요할 때 계정에 선택적으로 저장한다.
- 2026-07-15: 온라인 답변은 수락·거절 선택을 필수 답변으로 인정하고, 본문은 희망자만 남기는 선택적 `추가 의견`으로 취급한다.

## 1. 배경

현재 건넴은 React 상태와 브라우저 localStorage를 사용하는 개인용 웹앱이다. 한 기기에서 카드를 작성해 상대방에게 직접 보여주고, 제출된 답변을 같은 브라우저에 보관한다.

온라인 확장의 목표는 다음 두 가지다.

1. 아는 사람에게 링크를 공유해 여러 답변을 수집한다.
2. 불특정 다수가 카드를 발견하고 참여할 수 있는 공개 소통 영역으로 확장한다.

첫 번째 목표를 온라인 MVP로 우선 구현하고, 두 번째 목표는 운영·안전 기능을 갖춘 뒤 단계적으로 확장한다.

## 2. 제품 원칙

- 건넴의 중심 정보는 작성한 질문보다 `받은 답변`이다.
- 질문 작성과 카드 열람만으로는 답변 히스토리를 만들지 않는다.
- 수락 또는 거절을 선택해 제출한 순간 답변 기록을 생성한다.
- 답변 본문은 필수 답변이 아니라 선택적인 `추가 의견`이다.
- 카드 접근 범위와 받은 답변 공개 범위는 서로 독립적으로 관리한다.
- 응답자는 제출 전에 자신의 선택·닉네임·답변이 누구에게 보이는지 확인할 수 있어야 한다.
- 공개 범위를 변경하더라도 기존 응답자의 공개 기대를 침해하지 않는다.

## 3. 카드 작성 전 건네는 방식 선택

새 카드 작성을 시작할 때 다음 세 가지 중 하나를 선택한다.

```js
deliveryMode: "direct";
// direct | link | public
```

### 3.1 직접 건넴 (`direct`)

- 현재의 로컬 기반 사용 방식을 유지한다.
- 같은 기기에서 상대방에게 카드를 직접 보여준다.
- 서버 저장과 공유 링크를 사용하지 않는다.
- 카드와 답변은 현재 브라우저의 localStorage에 저장한다.
- 로그인 없이 사용할 수 있다.

사용자 안내:

> 지금 사용하는 기기에서 상대방에게 직접 카드를 보여줘요.  
> 카드와 답변은 이 브라우저에만 저장돼요.

작성 완료 행동: `상대에게 보여주기`

### 3.2 링크로 건넴 (`link`)

- 카드와 받은 답변을 서버에 저장한다.
- 공유 링크를 생성한다.
- 작성자 로그인 없이 카드를 만들 수 있다.
- 작성자에게 응답 링크와 별도의 비밀 관리 링크를 발급한다.
- 관리 링크로 답변 확인, 설정 변경, 마감, 링크 재발급과 기록 삭제를 수행한다.
- 원하는 경우 카드를 계정에 저장해 관리 링크 분실 복구와 여러 기기 관리를 지원한다.
- 링크를 받은 사람만 카드를 발견할 수 있다.
- `모두의 건넴`과 검색 결과에는 표시하지 않는다.
- 여러 응답자의 답변을 작성자 온라인 답변함에 모은다.

사용자 안내:

> 공유 링크를 받은 사람만 카드를 보고 답변할 수 있어요.

작성 완료 행동: `공유 링크 만들기`

### 3.3 모두에게 건넴 (`public`)

- 카드와 받은 답변을 서버에 저장한다.
- 공유 링크를 생성한다.
- `모두의 건넴`에서 누구나 카드를 발견하고 답변할 수 있다.
- 공개 콘텐츠의 소유권과 신고·제재 처리를 위해 작성자 로그인이 필요하다.
- 향후 공개 투표와 커뮤니티 기능의 기반으로 사용한다.

사용자 안내:

> 건넴을 사용하는 누구나 카드를 찾아보고 답변할 수 있어요.

작성 완료 행동: `모두에게 건네기`

## 4. 공개 카드 영역의 명칭

사용자 화면에서는 모호한 `공개 목록`이라는 표현을 사용하지 않는다.

- 공개 카드 공간 이름: `모두의 건넴`
- 진입 안내: `다른 사람들이 건넨 카드를 둘러보고 마음에 답해보세요.`
- 내부 구현 용어로만 `public` 또는 `public_listed`를 사용할 수 있다.

## 5. 건네는 방식 변경

- `direct → link`: 로그인 없이 서버에 복사하고 비밀 관리 링크를 발급할 수 있다.
- `direct → public`: 작성자 로그인 후 로컬 카드를 서버에 업로드한다.
- 비회원 `link → public`: 먼저 작성자 계정에 카드를 귀속한 뒤 `모두의 건넴`에 표시한다.
- `public → link`: `모두의 건넴`에서는 제거하지만 기존 공유 링크는 유지한다.
- 온라인 카드에서 직접 사용이 필요하면 서버 원본을 삭제하지 않고 로컬 복사본을 만든다.
- 기존 공유 링크의 비활성화·재발급은 건네는 방식과 별도 설정으로 다룬다.

## 6. 받은 답변 공개 범위

온라인 방식인 `link`와 `public`에서 다음 세 가지 중 하나를 선택한다. `direct`에는 이 설정을 적용하지 않는다.

```js
responseVisibility: "owner_only";
// owner_only | counts_only | all_responses
```

### 6.1 작성자만 보기 (`owner_only`)

- 작성자만 선택, 닉네임과 답변 내용을 확인한다.
- 다른 응답자에게 집계와 답변을 노출하지 않는다.

응답자 안내:

> 작성자만 답변을 확인할 수 있어요.  
> 닉네임과 답변 내용은 다른 사람에게 공개되지 않아요.

### 6.2 결과 숫자만 공개 (`counts_only`)

- 수락·거절 인원수만 공개한다.
- 닉네임과 답변 내용은 작성자만 확인한다.

응답자 안내:

> 수락·거절 결과 숫자는 다른 사람에게 공개돼요.  
> 닉네임과 답변 내용은 작성자만 확인할 수 있어요.

### 6.3 전체 답변 공개 (`all_responses`)

- 수락·거절 선택, 닉네임과 답변 내용을 공개한다.
- 다른 응답자도 공개된 답변 목록을 볼 수 있다.

응답자 안내:

> 제출한 선택, 닉네임과 답변 내용이 다른 사람에게 공개돼요.

## 7. 기본값과 설정 조합

권장 기본값은 다음과 같다.

| 건네는 방식 | 답변 공개 기본값 |
| --- | --- |
| 직접 건넴 | 해당 없음 |
| 링크로 건넴 | 작성자만 보기 |
| 모두에게 건넴 | 결과 숫자만 공개 |

`모두에게 건넴`도 닉네임과 답변 내용을 자동으로 전체 공개하지 않는다.

## 8. 공개 범위 변경과 응답자 동의

- 더 비공개인 방향으로의 변경은 즉시 적용할 수 있다.
- `owner_only → counts_only`는 개인 답변 내용을 노출하지 않으므로 즉시 적용할 수 있다.
- `owner_only/counts_only → all_responses` 변경 시 공개에 동의한 답변만 노출한다.
- 제출 당시 공개 범위와 공개 동의 여부를 답변별로 저장한다.

```js
response: {
  publicConsent: false,
  visibilityAtSubmission: "owner_only"
}
```

필요한 경우 제출 폼에 다음 동의 항목을 제공한다.

> 닉네임과 답변이 다른 사람에게 공개되는 것에 동의합니다.

## 9. 온라인 답변 저장 규칙

- 질문 작성: 기록하지 않음
- 카드 열람: 기록하지 않음
- 수락·거절 선택만 하고 제출하지 않음: 기록하지 않음
- 수락 또는 거절 선택 제출: 추가 의견이 없어도 새 기록 생성
- 수락·거절을 선택하지 않은 추가 의견 제출: 기록하지 않음
- 추가 의견이 공백뿐인 경우: 빈 값으로 정규화하고 선택 답변만 기록
- 답변 수정 제출: 기존 기록 갱신
- 답변 초기화 후 새 제출: 새로운 기록 생성

답변 카드는 다음 순서로 정보를 강조한다.

1. 수락·거절 선택
2. 상대방이 남긴 추가 의견이 있으면 해당 내용
3. 받은 시각과 닉네임
4. 펼쳐볼 수 있는 원래 질문과 결과 문구

추가 의견이 없는 기록에는 빈 문구나 빈 카드 영역을 만들지 않고 선택 결과를 중심으로 표시한다.

작성자는 받은 답변 기록을 확인 후 개별 삭제할 수 있다.

## 10. 작성자 관리 영역

온라인 작성자 화면은 다음 영역으로 나눈다.

- `내 카드`
- `받은 답변`
- `카드 설정`
- `공유 링크 관리`

비회원 링크 카드는 비밀 관리 링크로 위 영역에 접근한다. 계정에 저장한 카드와 `모두에게 건넴` 카드는 로그인한 작성자의 관리 영역에서 접근한다.

카드 설정의 우선 항목:

```text
건네는 방식
○ 링크로 건넴
○ 모두에게 건넴

받은 답변 공개 범위
○ 작성자만 보기
○ 결과 숫자만 공개
○ 전체 답변 공개
```

## 11. 제안 데이터 모델

```js
card: {
  id: "card-id",
  ownerId: null,
  publicToken: "share-token",
  managementTokenHash: "management-token-hash",
  claimedAt: null,
  deliveryMode: "link",
  responseVisibility: "owner_only",
  questionText: "오늘 저녁에 게임할래?",
  acceptText: "좋아",
  rejectText: "오늘은 어려워",
  createdAt: "...",
  updatedAt: "..."
}
```

```js
response: {
  id: "response-id",
  cardId: "card-id",
  nickname: "ㅇㅇ",
  responseType: "accept",
  replyText: null,
  publicConsent: false,
  visibilityAtSubmission: "owner_only",
  createdAt: "...",
  updatedAt: "..."
}
```

`direct` 카드는 현재 localStorage 모델을 사용하며, 온라인 전환 시 서버 모델로 복사한다.

`link` 카드를 계정에 저장하면 `ownerId`와 `claimedAt`을 기록한다. 계정 귀속 후 기존 관리 링크를 폐기할지 유지할지는 보안 설계에서 최종 확정한다.

## 12. 권한 원칙

- 로그인 작성자 또는 유효한 비밀 관리 토큰을 가진 비회원 작성자만 카드 수정, 설정 변경, 전체 답변 열람과 삭제가 가능하다.
- `모두에게 건넴`의 생성·게시·관리는 로그인 작성자만 가능하다.
- 응답자는 공유된 카드 열람과 답변 제출만 가능하다.
- `owner_only` 답변은 작성자 이외의 사용자에게 반환하지 않는다.
- `counts_only`에서는 집계값만 제공하고 닉네임과 답변 원문을 반환하지 않는다.
- `all_responses`에서도 공개 동의 조건을 만족한 답변만 반환한다.
- 권한 검사는 화면 숨김이 아니라 서버와 데이터베이스 정책에서 강제한다.

## 13. 단계별 업데이트 방향

### 1단계: 현재 로컬 모드 정리

- 새 카드 시작 시 세 가지 건네는 방식 선택 UI 설계
- 기존 직접 건넴 흐름을 `direct` 모드로 유지
- `링크로 건넴`은 비회원 관리 링크 발급, `모두에게 건넴`은 로그인 안내 연결

### 2단계: 링크 공유 MVP

- 비회원 카드 생성과 비밀 관리 링크
- 선택적인 계정 저장과 카드 귀속
- 카드 서버 저장
- 공유 링크 생성
- 수락·거절 필수 선택과 선택적 추가 의견 제출
- 작성자 전용 온라인 답변함
- 답변 공개 범위 적용

### 3단계: 운영과 안전

- 신고와 차단
- 스팸·자동 제출 방지와 요청 제한
- 관리자 검토 화면
- 콘텐츠 정책과 사용자 제재
- 개인정보·보관·삭제 정책

### 4단계: 모두의 건넴

- 공개 카드 탐색 화면
- `모두에게 건넴` 게시·해제
- 결과 숫자와 동의된 공개 답변 표시
- 검색, 분류와 정렬

## 14. 추가 설계가 필요한 항목

다음 항목은 이 RFC에서 확정하지 않는다.

- 닉네임 필수 여부와 익명 답변
- 추가 의견 최대 길이와 허용 문자 정책
- 한 사람의 중복 응답 허용 기준
- 답변 수정 가능 기간
- 답변 마감과 카드 만료
- 공유 링크 비활성화·재발급
- 선택적 계정 저장의 로그인 방식
- 계정 귀속 후 기존 비밀 관리 링크 유지·폐기 정책
- 공개 카드 카테고리·검색·추천
- 신고·차단·제재 세부 정책
- 이미지 서버 저장과 용량 제한
- 기존 localStorage 데이터의 계정 이전 정책

## 15. 승인 후 문서 반영

이 RFC의 방향이 확정되면 구현 전에 다음 작업을 수행한다.

1. `PRD.md`에 온라인 제품 범위와 인수 조건을 반영한다.
