# 건넴 에이전트 오케스트레이션 매핑

이 문서는 총괄 지휘자가 어떤 작업에 어떤 에이전트를 참여시키고, 산출물·결정·품질 게이트·인계를 어떻게 관리하는지 정의한다.

## 1. 오케스트레이션 원칙

- 모든 작업에 모든 에이전트를 호출하지 않는다.
- 제품 의미가 바뀌면 기획자를, 시스템 계약이 바뀌면 설계자를 먼저 호출한다.
- 프론트엔드와 백엔드는 승인된 계약 이후 독립 작업만 병렬로 진행한다.
- 품질 관리자는 설계 게이트, 구현 완료와 배포 전에 검토한다.
- 총괄 지휘자는 사용자 승인 상태, 문서, 코드와 `TODO.md`가 일치하도록 관리한다.
- RFC 초안은 사용자 승인과 PRD 반영 전까지 구현 명령이 아니다.

## 5. SQL 관리 규칙

- 백엔드 에이전트가 스키마, 인덱스, query template, migration과 rollback 안내를 관리한다.
- SQL 파일은 버전 번호를 붙이고 이미 적용된 migration을 수정하지 않는다.
- 운영 DB 적용 전 백업, 대상 버전, 대상 DB명과 적용 순서를 확인한다.
- 비밀번호, 원본 토큰과 실제 운영 개인정보를 SQL 파일에 넣지 않는다.
- raw share·management·edit token은 저장하지 않고 애플리케이션에서 만든 digest만 저장한다.
- rollback SQL은 데이터 손실 위험을 명시하고 사용자 승인 없이 실행하지 않는다.
- phpMyAdmin 수동 실행 결과도 migration 기록과 검증 로그에 남긴다.
- migration 파일의 SHA-256, 실행 환경, 실행자, 백업과 결과를 배포 기록에 남긴다.

## 5.1 환경·사고·복구 책임

| 관리 항목 | 주 담당 | 승인·검토 |
| --- | --- | --- |
| dev/staging/prod 분리 | 총괄·백엔드 | 품질 |
| DB migration ledger·checksum | 백엔드 | 설계자·품질 |
| DB 백업·복원 리허설과 RTO 기록 | 백엔드 | 품질·사용자 승인 |
| 비밀키·토큰 유출 대응과 회전 | 백엔드 | 총괄·품질 |
| 배포 rollback 실행 | 총괄 | 백엔드·품질·사용자 승인 |
| 테스트·품질 증거 보관 | 품질 | 총괄 |

운영 증거의 실제 저장 위치는 배포 환경 설계 시 확정하며, 개인정보와 비밀값이 포함된 로그는 저장소에 커밋하지 않는다.

## 6. 회의 운영

### 제안 라운드

1. 총괄이 목표, 결정 등급과 산출물을 지정한다.
2. 기획·설계·구현 담당은 각자 독립 의견을 제출한다.
3. 의견은 `합의`, `충돌`, `보류`, `사용자 결정`으로 분류한다.

### 반박·검증 라운드

1. 프론트엔드와 백엔드가 구현 가능성을 교차 검토한다.
2. 품질 관리자가 개인정보·보안·접근성·회귀 관점에서 심사한다.
3. 총괄이 권고안을 작성하되 D4·D5를 대신 확정하지 않는다.

### 사용자 보고

- 합의된 방향
- 의견이 갈린 항목과 근거
- 에이전트 권고안
- 사용자 선택이 필요한 항목
- 승인 후 시작할 단계

## 7. 결정 관리

| 등급 | 결정 주체 | 기록 위치 |
| --- | --- | --- |
| D1 구현 세부 | 담당 에이전트 | 코드·인계 |
| D2 UX·기본값 | 기획자 제안, 총괄 조정 | PRD 또는 UI 규칙 |
| D3 API·데이터 계약 | 설계자 제안, 기술 검토 | 아키텍처·OpenAPI |
| D4 제품·개인정보·운영 | 사용자 | RFC 승인 기록·PRD |
| D5 외부 실행·비용·배포 | 사용자 | 실행 기록·배포 문서 |

결정 기록에는 날짜, 결정자, 선택지, 선택 결과, 영향받는 문서와 재검토 조건을 남긴다.

## 8. 인계와 상태 관리

각 작업은 다음 정보를 포함해 인계한다.

```md
- 목표
- 완료 결과
- 변경 파일·계약
- 검증 증거
- 결정과 가정
- 미결정 사항
- 위험·롤백 지점
- 다음 담당과 선행 조건
```

총괄은 인계 후 `TODO.md`와 계획 상태를 갱신한다. 코드가 작성됐다는 이유만으로 `verified` 또는 `done`으로 표시하지 않는다.

## 9. 중단·보고 조건

- 사용자 데이터 공개 범위가 달라질 때
- 기존 direct 동작 또는 localStorage 데이터가 바뀔 때
- 외부 서비스, 비용, 계정, 비밀키가 필요할 때
- 운영 DB migration, 데이터 삭제, 배포 직전
- 품질 관리자가 FAIL 또는 BLOCKED로 판정할 때
- 역할 간 결론이 제품 결과를 실질적으로 다르게 만들 때
