# 에이전트 공통 프로토콜

## 1. 공통 목표

모든 에이전트는 사용자가 승인한 건넴의 사용자 가치와 범위를 구현하면서 기존 동작, 데이터, 문서와 사용자의 직접 수정 사항을 보호한다.

## 2. 작업 시작 루프

모든 작업은 다음 순서로 시작한다.

1. 사용자의 최신 요청을 한 문장 목표로 재정의한다.
2. `PRD.md`, 관련 `docs`, `AGENTS.md`, `TODO.md`를 읽는다.
3. 실제 코드를 조사해 문서와의 차이를 확인한다.
4. 필요한 역할, 선행 결정, 승인 게이트를 판정한다.
5. 위험이 낮고 범위가 명확하면 실행하고, 제품 방향을 바꾸는 미결정 사항은 사용자에게 요청한다.

## 3. 작업 상태

각 작업은 다음 상태 중 하나를 가진다.

```text
proposed → ready → in_progress → review → verified → done
                       └──────────────→ blocked
```

- `proposed`: 아이디어 또는 RFC 단계
- `ready`: 요구사항과 선행 조건이 충족됨
- `in_progress`: 구현 또는 문서화 진행 중
- `review`: 담당 역할의 작업이 끝나 검토 대기
- `verified`: 인수 조건과 검증 통과
- `done`: 문서와 TODO까지 반영 완료
- `blocked`: 외부 결정이나 권한 없이는 진행 불가

## 4. 의사결정 등급

| 등급 | 예시 | 처리 방식 |
| --- | --- | --- |
| D1 구현 세부 | 함수명, 내부 파일 분리 | 담당 에이전트가 결정하고 기록 |
| D2 사용자 경험 | 문구, 흐름, 기본값 | 기획자 제안, 총괄 조정, 필요 시 사용자 승인 |
| D3 시스템 계약 | API, 데이터 모델, 권한 정책 | 설계자 제안, 백엔드·품질 검토, 승인 후 구현 |
| D4 제품·운영 정책 | 공개 범위, 개인정보, 과금, 신고 | 사용자 승인 필수 |
| D5 외부 변경 | 배포, 운영 DB 변경, 외부 메시지 | 명시적 실행 승인 필수 |

## 5. 공통 인계 형식

각 에이전트는 작업 종료 시 다음 형식으로 인계한다.

```md
## 인계
- 목표:
- 완료한 결과:
- 변경한 파일/계약:
- 검증 증거:
- 결정 사항:
- 미결정 사항:
- 위험 및 회귀 지점:
- 다음 담당자:
```

## 6. 충돌 해결

1. 사용자 최신 지시가 최우선이다.
2. 승인된 `PRD.md`가 RFC보다 우선한다.
3. 제품 의미 충돌은 기획자가 정리하고 총괄 지휘자가 사용자 결정을 요청한다.
4. 기술 계약 충돌은 설계자가 대안을 비교하고 총괄 지휘자가 결론을 관리한다.
5. 품질 관리자는 인수 조건이나 보안 기준을 충족하지 못한 완료 선언을 거부할 수 있다.

## 7. 안전 및 금지 규칙

- 승인되지 않은 RFC를 구현 범위로 간주하지 않는다.
- 비밀키, 개인정보, 운영 데이터가 로그나 저장소에 노출되지 않게 한다.
- 사용자 승인 없이 배포, 운영 DB 마이그레이션, 데이터 삭제를 실행하지 않는다.
- 클라이언트 UI 숨김만으로 권한을 보호했다고 판단하지 않는다.
- 테스트하지 않은 항목을 완료로 표시하지 않는다.
- 다른 에이전트나 사용자의 관련 없는 변경을 되돌리지 않는다.

## 8. 완료 정의

작업은 다음을 모두 만족할 때 완료다.

- 승인된 요구사항과 인수 조건을 충족한다.
- 관련 빌드·테스트·수동 시나리오를 검증했다.
- 접근성, 보안, 개인정보 영향이 검토됐다.
- 관련 문서, 매핑, `TODO.md`가 실제 상태와 일치한다.
- 미결정 사항과 알려진 제한이 사용자에게 명확히 전달됐다.
