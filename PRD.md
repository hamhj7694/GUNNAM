# 건넴 구현 기반 PRD

## 1. 제품 개요

건넴은 말로 꺼내기 어려운 질문이나 제안을 카드 형태로 작성해 상대에게 보여주고, 상대가 수락 또는 거절을 선택하도록 돕는 모바일 중심 웹 애플리케이션이다.

한 기기와 한 브라우저 세션 안에서 다음 과정을 지원한다.

1. 작성자가 메인 질문 카드와 수락·거절 결과 화면을 만든다.
2. 작성자가 완성된 카드를 상대에게 직접 보여준다.
3. 상대가 수락 또는 거절 버튼을 누른다.
4. 선택에 맞는 결과 메시지와 선택적 이미지를 보여준다.
5. 작성자가 허용한 경우 상대가 짧은 답장을 남긴다.

현재 제품은 링크 공유나 서버 전송 서비스가 아니라, React 상태와 브라우저 localStorage를 이용하는 단일 기기 MVP다.

## 2. 제품 목표

- 질문이나 제안을 짧고 명확한 카드로 전달한다.
- 상대가 수락과 거절 중 하나를 부담 없이 선택하게 한다.
- 선택 이후의 메시지, 이미지, 답장 여부를 작성자가 미리 구성하게 한다.
- 데스크톱에서도 모바일 앱과 유사한 좁은 프레임으로 제공하고, 작은 화면에서는 전체 화면으로 제공한다.

## 3. 사용자와 핵심 시나리오

### 작성자

- 빈 카드 작성을 시작한다.
- 메인 문구와 선택적 이미지를 입력한다.
- 수락·거절 버튼 문구를 직접 정한다.
- 수락 및 거절 결과 문구와 선택적 이미지를 설정한다.
- 각 결과 화면에서 답장을 받을지 설정한다.
- 최종 카드를 상대에게 보여준다.

### 응답자

- 메인 카드의 문구와 선택적 이미지를 확인한다.
- 수락 또는 거절을 선택한다.
- 선택 결과 화면을 확인한다.
- 답장 기능이 켜져 있으면 메시지를 제출한다.

## 4. 정보 구조와 화면 흐름

```text
홈 (/)
  → 카드 작성 (/create)
      1단계: 메인 카드
      2단계: 수락 페이지
      3단계: 거절 페이지
  → 상대에게 보여주는 카드 (/show)
      → 수락 결과 (/result/accept)
      → 거절 결과 (/result/reject)
  → 받은 답변 히스토리 (/history)
```

별도의 미리보기 페이지는 없다. `/create`의 각 단계에 실제 화면과 유사한 편집형 미리보기가 포함된다. 정의되지 않은 경로는 홈으로 대체 이동한다.

## 5. 기능 요구사항

### 5.1 홈

- 서비스명, 짧은 소개, 로그인 없이 사용할 수 있다는 안내를 표시한다.
- `새 카드 만들기`를 누르면 로컬 카드 데이터를 초기값으로 재설정하고 `/create`로 이동한다.
- `기존 카드 만들기`를 누르면 저장된 작성 상태를 유지한 채 `/create`로 이동한다.
- `카드 히스토리`를 누르면 제출된 답변을 모아보는 `/history`로 이동한다.
- 샘플 카드로 시작하는 기능은 제공하지 않는다.

### 5.2 카드 작성

작성 화면은 메인 카드, 수락 페이지, 거절 페이지의 3단계로 구성한다.

- 상단 탭을 눌러 어느 단계로든 즉시 이동할 수 있다.
- `이전`은 이전 단계로 이동하며, 첫 단계에서는 홈으로 이동한다.
- `다음`은 현재 입력의 유효성 검사 없이 다음 단계로 이동한다.
- 마지막 단계의 `보여주기`에서만 메인 문구 필수 여부를 검사한다.
- 메인 문구가 비어 있으면 첫 단계로 이동하고 오류를 표시한다.

#### 메인 카드 단계

- 메인 문구를 카드 내부 textarea에서 직접 입력한다.
- 메인 이미지는 선택 사항이다.
- 수락·거절 버튼 문구를 버튼 모양 입력란에서 수정한다.
- 이미지가 있으면 카드 안에 즉시 반영한다.

#### 수락 페이지 단계

- 수락 결과 문구를 결과 카드 내부에서 직접 입력한다.
- 수락 이미지 또는 QR 이미지를 선택적으로 추가한다.
- 수락 후 답장 입력 노출 여부를 토글한다.
- 답장이 켜져 있으면 비활성화된 답장 폼을 미리보기로 표시한다.

#### 거절 페이지 단계

- 거절 결과 문구를 결과 카드 내부에서 직접 입력한다.
- 거절 이미지를 선택적으로 추가한다.
- 거절 후 답장 입력 노출 여부를 토글한다.
- 답장이 켜져 있으면 비활성화된 답장 폼을 미리보기로 표시한다.

### 5.3 이미지 선택

- 브라우저 파일 선택기에서 `image/*` 파일을 선택한다.
- `FileReader`로 파일을 data URL로 변환해 React 상태에 저장한다.
- 이미지 선택 후 선택 완료 상태, 다시 선택, 삭제 기능을 제공한다.
- 서버 업로드는 수행하지 않는다.
- 이미지가 없으면 실제 카드와 결과 화면에 placeholder나 빈 이미지 영역을 렌더링하지 않는다.
- 메인 및 결과 이미지는 `object-fit: contain`으로 최대한 잘리지 않게 표시한다.

### 5.4 상대에게 보여주는 카드

- 메인 이미지가 있으면 이미지와 문구를, 없으면 문구만 표시한다.
- 수락·거절 버튼 외 편집 UI를 표시하지 않는다.
- 수락을 누르면 `/result/accept`, 거절을 누르면 `/result/reject`로 이동한다.
- 좌측 상단 뒤로가기 버튼은 `/create`로 이동한다.

### 5.5 수락 및 거절 결과

두 결과 화면은 동일한 동작 구조를 가지며 색상 맥락만 다르다.

- 설정된 결과 이미지가 있을 때만 표시한다.
- 계산된 결과 메시지를 표시한다.
- 해당 결과의 답장 기능이 켜져 있을 때만 답장 입력 폼을 표시한다.
- 공백뿐인 답장은 제출할 수 없고 오류 메시지를 표시한다.
- 제출 시 앞뒤 공백을 제거한 답장을 전역 상태에 저장한다.
- 제출된 답장은 입력 폼 위에 `ReplyCard`로 표시한다.
- 우측 상단의 원형 `답변 초기화` 버튼은 현재 결과의 답장 텍스트, 제출 상태와 히스토리 연결 ID만 지운다.
- 좌측 상단 뒤로가기 버튼은 `/show`로 이동한다.

### 5.6 받은 답변 히스토리

- 질문 작성이나 결과 선택만으로는 기록하지 않고, 유효한 답변을 `남기기`로 제출한 순간에만 기록한다.
- 받은 답변을 카드의 중심 정보로 표시하고 수락·거절 구분과 받은 시각은 보조 정보로 표시한다.
- 질문과 결과 문구는 기본적으로 숨기고 `어떤 카드에 대한 답변인지 보기`에서 펼쳐 표시한다.
- 답변 수정 제출은 같은 기록을 갱신하고, 답변 초기화 후 다음 제출은 새 기록을 만든다.
- 히스토리는 새 카드 초기화와 분리해 보존하며 각 기록은 확인 후 개별 삭제할 수 있다.

## 6. 표시 및 fallback 규칙

| 항목 | 저장값이 있을 때 | 비어 있을 때 |
| --- | --- | --- |
| 수락 버튼 | 입력 문구 | `좋아요` |
| 거절 버튼 | 입력 문구 | `싫어요` |
| 수락 결과 | 수락 결과 문구 | 최종 수락 버튼 문구 |
| 거절 결과 | 거절 결과 문구 | 최종 거절 버튼 문구 |
| 이미지 | 이미지 표시 | 요소 자체를 렌더링하지 않음 |

문구의 비어 있음은 `trim()` 결과를 기준으로 판단한다. 결과 문구는 줄바꿈과 공백을 보존해 표시한다.

## 7. 상태 모델

`App`이 모든 카드 데이터를 React 상태로 보유하고 각 페이지에 전달하며, 변경된 원본 상태를 브라우저 localStorage에 동기화한다.

```js
const cardData = {
  mainText: "",
  mainImage: null,
  acceptButtonText: "",
  rejectButtonText: "",
  acceptImage: null,
  acceptResultText: "",
  acceptReplyEnabled: true,
  acceptReplyText: "",
  acceptReplySubmitted: false,
  acceptReplyHistoryId: null,
  rejectImage: null,
  rejectResultText: "",
  rejectReplyEnabled: true,
  rejectReplyText: "",
  rejectReplySubmitted: false,
  rejectReplyHistoryId: null
};
```

- 부분 업데이트는 기존 상태에 새 필드를 병합한다.
- 표시용 버튼 및 결과 문구는 `getCardDisplay`에서 파생한다.
- 답장 기능은 수락과 거절에 각각 독립된 상태를 가진다.
- 앱 시작 시 localStorage의 저장 데이터를 초기 상태와 병합해 복구한다.
- 카드 작성 중 문구, 이미지 data URL, 답장 설정은 변경될 때마다 같은 브라우저에 자동 저장된다.
- 홈의 `새 카드 만들기`는 메모리와 로컬 저장 데이터를 초기화하고, `기존 카드 만들기`는 저장 데이터를 유지한다.
- 서버 저장이나 다른 기기와의 동기화는 제공하지 않는다.

## 8. UI 및 반응형 요구사항

- 기본 배경 위 중앙에 최대 430px 너비의 모바일 프레임을 표시한다.
- 프레임은 둥근 모서리, 옅은 테두리와 그림자를 사용한다.
- 420px 이하에서는 프레임의 테두리·둥근 모서리·그림자를 제거하고 화면 전체를 사용한다.
- 메인 색상은 녹색이며 수락 결과는 파란색, 거절 결과는 붉은색 맥락을 사용한다.
- 카드와 결과 영역은 점선 내부 테두리로 구분한다.
- 긴 문구와 줄바꿈이 카드 폭을 넘지 않도록 줄바꿈한다.
- 버튼과 입력 요소는 모바일 터치에 적합한 크기를 유지한다.

## 10. 범위 밖 기능

현재 구현에는 다음 기능이 없다.

- 회원가입, 로그인, 사용자 계정
- 서버 저장, 데이터베이스, 다른 기기 동기화
- 카드 링크 생성, 공유, 다른 기기로 전달
- 이미지 서버 업로드
- 위치 정보
- 실시간 채팅, 알림, 통계
- 거절 사유 분류나 다시 묻기 흐름
- 접근 권한이나 작성자·응답자 역할 검증

## 11. 구현 기준 인수 조건

- 홈의 두 카드 만들기 버튼은 각각 초기화 또는 기존 작성 상태 유지 후 작성 화면으로 이동한다.
- 카드 작성 중 새로고침하면 같은 브라우저의 저장 데이터가 복구된다.
- 작성 단계는 탭과 이전·다음 버튼으로 이동할 수 있다.
- 최종 보여주기 시 메인 문구가 비어 있으면 상대 화면으로 이동하지 않는다.
- 빈 수락·거절 버튼 문구는 각각 기본 문구로 표시된다.
- 빈 결과 문구는 해당 버튼의 최종 표시 문구를 사용한다.
- 이미지가 없는 모든 화면에는 빈 이미지 박스가 남지 않는다.
- 수락과 거절 선택이 각각 올바른 결과 경로로 이동한다.
- 수락 및 거절 답장 토글은 서로 독립적으로 동작한다.
- 답장 토글이 꺼져 있으면 해당 결과 화면에 입력 폼이 없다.
- 빈 답장은 제출되지 않고, 유효한 답장은 카드로 표시된다.
- 수락·거절 결과의 답변 초기화는 해당 결과의 답변만 지우고 카드 작성 데이터와 반대 결과 답변은 유지한다.
- 유효한 답변 제출만 히스토리에 기록되고 수정 제출은 기존 기록을 갱신한다.
- 히스토리에서 질문은 보조 맥락으로 제공되며 개별 답변 기록을 확인 후 삭제할 수 있다.
- 새 카드 초기화 후에도 기존 받은 답변 히스토리는 유지된다.
- 알 수 없는 URL은 홈으로 이동한다.

## 12. 현재 저장소에서 확인된 제약과 위험

- 의존성 설치 디렉터리가 없어 현재 환경에서는 실제 빌드 완료까지 검증되지 않았다.
- 일부 소스의 한글 문자열이 깨진 문자로 저장되었거나 표시되는 정황이 있다. 사용자 노출 문구와 JSX 구문을 UTF-8 기준으로 별도 점검해야 한다.
- 직접 URL 진입 시 저장된 카드가 없을 때의 별도 데이터 가드는 없다.
- 업로드 이미지의 파일 크기·형식·해상도 검증이 없으며 큰 data URL은 메모리와 localStorage 용량을 많이 사용할 수 있다.
- 접근성은 일부 `aria-label`과 기본 HTML 요소를 사용하지만, 키보드 흐름과 스크린리더 동작에 대한 별도 검증은 확인되지 않았다.
